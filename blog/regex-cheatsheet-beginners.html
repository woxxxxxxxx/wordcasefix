Regex Cheatsheet: Patterns Every Developer Should Know

By WordCaseFix  ·  9 min read  ·  Programming

Regular expressions are one of those skills that pay back the investment many times over. Once you can write a precise pattern, tasks that would otherwise require ten lines of imperative code collapse to one. The bad news: the syntax is dense and unforgiving, and most regex tutorials drown you in edge cases before you have a working mental model. This cheatsheet focuses on the patterns you will actually use weekly, with worked examples you can paste into a tester and adapt.

Every example below uses standard PCRE/ECMAScript syntax — the dialect supported by JavaScript, Python's re, PHP, and most modern languages. Where a feature differs between flavors, that is noted explicitly.

The Building Blocks: Anchors, Classes, Quantifiers

A regex is built from three kinds of pieces: things that match a specific character or set, anchors that match positions rather than characters, and quantifiers that say "how many." Once you internalize this trio you can read almost any pattern.

TokenMeaning
.Any character except newline
\d / \DDigit / non-digit
\w / \WWord char (a-z A-Z 0-9 _) / non-word
\s / \SWhitespace / non-whitespace
[abc]Any one of a, b, or c
[^abc]Any character except a, b, c
[a-z]Range a through z
^ / $Start / end of string (or line in multiline mode)
\bWord boundary (zero-width)
*0 or more (greedy)
+1 or more (greedy)
?0 or 1 (optional)
{n,m}Between n and m repetitions
*? +?Non-greedy versions

Anchors are the most common source of confusion for beginners. ^abc does not mean "no abc" — that is [^abc] inside a character class. ^ outside a character class means "start of input."

Groups, Alternation, and Backreferences

Parentheses serve two purposes: grouping for quantification and capturing for later use. (abc)+ matches one or more occurrences of the literal sequence "abc." The captured text is available as group 1.

Use (?:...) for non-capturing groups when you only need grouping, not capture — it's slightly faster and keeps your group numbering clean. Named groups ((?<name>...)) make complex patterns much more readable.

Named groups example — parsing a date:
Pattern: (?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})
Input: 2025-06-18
Result: year=2025, month=06, day=18

Alternation | means "or." cat|dog matches either word. Note that alternation has very low precedence: ^cat|dog$ means "starts with cat OR ends with dog," not "starts and ends with cat or dog." Wrap with a group: ^(cat|dog)$.

Lookarounds: Match Without Consuming

Lookarounds are zero-width assertions — they check that something is or isn't present without including those characters in the match. There are four:

  • (?=...) — positive lookahead: "followed by..."
  • (?!...) — negative lookahead: "not followed by..."
  • (?<=...) — positive lookbehind: "preceded by..."
  • (?<!...) — negative lookbehind: "not preceded by..."

The classic example is password validation. Suppose you require at least 8 characters, one digit, and one uppercase. A single pattern handles this cleanly:

Password rule with lookaheads:
^(?=.*\d)(?=.*[A-Z]).{8,}$

Read it as: "from start, ahead is a digit somewhere, ahead is an uppercase somewhere, then 8+ of anything, then end."

Common Patterns Worth Memorizing

These are the patterns I find myself reaching for most weeks. Test each one against your own data before deploying — every real-world dataset has surprises.

TaskPattern
Simple email^[\w.+-]+@[\w-]+\.[\w.-]+$
URL (http/https)^https?://[\w.-]+(:\d+)?(/[^\s]*)?$
ISO date (YYYY-MM-DD)^\d{4}-(0[1-9]|1[0-2])-(0[1-9]|[12]\d|3[01])$
Strong password (8+, mixed)^(?=.*[a-z])(?=.*[A-Z])(?=.*\d).{8,}$
IPv4 (loose)^(\d{1,3}\.){3}\d{1,3}$
Hex color^#([A-Fa-f0-9]{6}\|[A-Fa-f0-9]{3})$
UUID v4^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$
Trim whitespace^\s+|\s+$
Slug (kebab-case)^[a-z0-9]+(-[a-z0-9]+)*$

Note the email pattern is "simple" — a fully RFC-5322-compliant email regex is several thousand characters and rarely worth it. In practice, validate format loosely and confirm by sending a verification email.

Performance trap: patterns like (a+)+b can trigger catastrophic backtracking against input like aaaaaaaaa with no trailing b. Avoid nested unbounded quantifiers, and prefer possessive quantifiers (a++) or atomic groups ((?>...)) where supported.

Flags That Change Everything

Regex flags modify how the entire pattern is interpreted, and forgetting them is a top source of confusion. The four most important flags across languages are case insensitivity, multiline, dotall, and global. Case insensitivity makes [a-z] equivalent to [a-zA-Z] — convenient, but slower and sometimes wrong (Turkish "I" famously breaks naive case folding). Multiline mode changes the meaning of ^ and $ from "start/end of string" to "start/end of each line," which matters whenever you process multi-line text. Dotall mode lets . match newlines, useful for parsing blocks that span lines. Global mode (in JavaScript) controls whether .replace swaps the first match or all of them.

One practical recommendation: explicitly set every flag your pattern depends on, even when the default would work. Future maintainers — including future-you — will appreciate the clarity. A pattern that silently relies on the default of the dotall flag becomes a Heisenbug when copied between languages with different defaults.

Frequently Asked Questions

Should I parse HTML with regex?

For one-off extraction of a known tag in known input, regex is fine. For anything resembling real HTML parsing, use an actual HTML parser. HTML is not a regular language — comments, CDATA, and nested tags will defeat any non-trivial regex. Tools like HTML strippers use proper parsers under the hood for this reason.

What's the difference between greedy and lazy matching?

Greedy quantifiers (*, +) match as much as possible, then back off if the rest of the pattern doesn't fit. Lazy quantifiers (*?, +?) match as little as possible and expand only when forced. Use lazy when extracting content between delimiters, like "(.*?)" to capture the smallest string between quotes.

How do I match a literal special character like a dot?

Escape it with a backslash: \. matches a literal period. The characters needing escape are . ^ $ * + ? ( ) [ ] { } | \. Inside a character class, fewer characters are special — usually only ] \ ^ -. When in doubt, test in a regex tester before shipping the code.
